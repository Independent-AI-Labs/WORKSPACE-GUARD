Sandlock: Confining AI Agent Code with Unprivileged Linux Primitives

Cong Wang Multikernel Technologies, Inc.USA cwang@multikernel.io and Yusheng Zheng University of California, Santa CruzUSA yzhen165@ucsc.edu
Abstract.

AI agents increasingly run untrusted code on developer machines: shell commands generated by language models, third-party scripts retrieved at runtime, and tool plugins of unknown provenance. Existing isolation mechanisms impose tradeoffs that fit this workload poorly: containers and microVMs add privilege, image-management, and startup costs, while ad-hoc process controls and wrappers (e.g. chroot, ulimit) provide weak guarantees and little syscall-level control. Sandlock is a lightweight Linux process sandbox organized around a simple split: static, input-independent policy is compiled into kernel-enforced rules, while a narrow supervisor handles runtime-dependent decisions and virtualized effects. This split lets Sandlock enforce filesystem, network, IPC, and syscall policies without root, cgroups, images, or mandatory namespaces. It also supports dynamic network decisions, HTTP-level access control, TOCTOU-safe inspection of execve arguments, and reversible filesystem effects. On our workstation, Sandlock adds roughly 5 ms of startup overhead and runs Redis at bare-metal throughput (within measurement noise); its pipeline operator further supports per-stage confinement for separating data, network, and untrusted-content capabilities. Sandlock is available at https://github.com/multikernel/sandlock.

1. Introduction

AI coding assistants and autonomous agents (Yang et al., 2024a; Wang et al., 2025; Anthropic, 2025) routinely execute code that no human has reviewed: shell commands proposed by the model, scripts downloaded from package registries, build steps invoked through MCP-style tool servers (Model Context Protocol, 2026), and plug-ins from third-party marketplaces. Each invocation is a potential supply-chain exposure on the developer’s workstation: an unintended rm -rf, access to an SSH key, a network call to an attacker-controlled host, or a curl | sh pipeline invoked by a model-generated build script.

In principle, the operating system has the building blocks to confine such code. In practice, off-the-shelf options make tradeoffs that fit this workload poorly. Containers (Merkel, 2014) and microVMs (Agache et al., 2020) provide familiar isolation boundaries, but they bring image management, root or rootless configuration (Kerrisk, 2013; Docker Inc., 2026), KVM requirements, and startup costs that are visible when an agent runs many short-lived commands. gVisor (Google, 2024) interposes a user-mode kernel, improving isolation at the cost of compatibility and syscall overhead for build tools. Ad-hoc controls such as ulimit, privileged chroot, and launcher tools such as firejail/bubblewrap (Firejail Project, 2026; Bubblewrap Project, 2026) reduce deployment cost but offer limited syscall control, weak filesystem semantics, and no host- or HTTP-level network policy.

This workload needs a tool tailored to the agent workstation: unprivileged, low-latency, filesystem-aware, network-aware at the level of hosts and HTTP routes, and programmable so that a host application can enforce policies that depend on the program’s runtime behavior (e.g. revoke network access once the workload transitions from setup to running untrusted code, a transition observable at execve).

The design problem is not the absence of Linux isolation primitives, but deciding which policies should be enforced where. Static invariants, such as readable path prefixes, writable workspaces, TCP ports, IPC scope, and unconditional syscall denials, should stay in kernel-enforced rules. Policies that depend on runtime values, such as the resolved destination of a connect, the argv of an execve, an HTTP method/path, or a copy-on-write filesystem effect, require controlled supervisor involvement. Sandlock makes this split explicit.

Sandlock is a Rust process sandbox that realizes this split using Landlock (Salaün, 2017) for static filesystem, TCP-port, and IPC scope; seccomp-bpf (Drewry and Edge, 2024) for unconditional syscall filtering; and seccomp user notification (Kerrisk, 2024) with pidfd_getfd (Brauner, 2024) for runtime-dependent decisions and on-behalf operations.

Rather than exposing these mechanisms independently, Sandlock presents a single policy surface: static rules are pushed into Landlock and seccomp-bpf whenever possible, and only runtime-dependent operations reach the supervisor. The same split supports HTTP-level network policy, reversible filesystem effects, and pipeline stages with different confinements. On our workstation, Sandlock adds ∼\sim5 ms of startup overhead (∼\sim6 ms wall, 44×\times faster end-to-end than Docker on our setup) and no measurable Redis throughput overhead (within run-to-run noise).

This paper makes the following contributions:

  • •

    A policy model and design for unprivileged agent sandboxing that separates static kernel-enforced policy from runtime-dependent supervisor decisions, while preserving a uniform configuration surface (Sections 3–4).

  • •

    A TOCTOU-safe runtime policy mechanism (policy_fn) that exposes syscall events, including argv for execve, to host code without violating kernel re-read semantics (Section 5.1).

  • •

    Two applications of the split enforcement model: a seccomp-driven copy-on-write workspace that captures filesystem writes for commit/abort/dry-run without mount namespaces or root, and a COW-fork primitive that sustains ∼\sim1,900 sandboxed forks/s for map-reduce workloads (Section 5.2).

  • •

    A preliminary evaluation showing ∼\sim5 ms startup overhead, bare-metal Redis throughput within measurement noise, and lower p99 latency than Docker (Section 6).

Sandlock is open source at https://github.com/multikernel/sandlock.

2. Motivation

2.1. The Agent Workstation Threat Model

These systems interleave LLM reasoning with shell commands: editing files, installing packages, running tests, and calling network APIs. We assume the sandboxed process may execute attacker-influenced code or consume attacker-controlled data, including package scripts, downloaded files, tool outputs, and generated commands. The host kernel, the Sandlock supervisor, and the embedding agent runtime are trusted; kernel vulnerabilities, side channels, privileged local attackers, and deliberate exhaustion of global kernel resources are out of scope. Five properties of this workload shape our threat model:

Untrusted code is short-lived and frequent.

A single agent task may invoke npm install, pytest, curl, or generated Python snippets dozens of times per minute. Per-invocation startup overhead in the hundreds of milliseconds (typical of containers) is visible in human-facing latency.

The host is a developer machine.

Sandboxes run alongside the developer’s editor, SSH keys, and source trees. Confinement must be unprivileged (developer accounts rarely have free use of sudo) and must default-deny the rest of the home directory.

Network access is essential but dangerous.

Agents legitimately need to call LLM APIs, package registries, and documentation servers, but exfiltration to attacker-controlled hosts must be blocked. Allowlisting at the IP/port layer is necessary but not sufficient: a model that legitimately calls api.openai.com:443 can also call attacker-controlled endpoints through the same allowed IP. Useful policies often live at the HTTP method, host, and path level.

Agent tasks compose stages of asymmetric trust.

A growing class of agent failures comes not from untrusted code but from untrusted data: prompt injection (Debenedetti et al., 2025), where instructions embedded in a file, web page, or tool output hijack the agent’s own LLM. The danger materializes when a single context combines access to private data, the ability to communicate externally, and exposure to untrusted content: the “lethal trifecta” (Willison, 2025). Process-level confinement of one sandbox cannot break this combination, because a legitimate agent step often needs all three. This motivates splitting a task into differently confined stages: the stage exposed to untrusted content has no network, and the stage with network never sees it. Dual-LLM and capability-separation patterns (Willison, 2023; Debenedetti et al., 2025) prescribe this decomposition but assume an enforcement substrate that unprivileged sandboxes do not provide.

Agent behavior is generated, not known in advance.

A static policy must be fixed before the agent decides what to run, yet the safe policy often depends on information that exists only at runtime. An npm install step legitimately needs network access and writes under node_modules that the test step it sets up should not retain; the transition between the two is observable (the execve of the test runner) but not expressible as a single static ruleset over one invocation. The decision can also hinge on a syscall argument (the argv of a model-generated command, the resolved destination of a connect) that no allowlist could enumerate ahead of time. Finally, the agent runtime embedding the sandbox usually knows more than the static config did: which tool is being invoked, what the previous step did, whether this sub-task was ever supposed to spawn a process at all. Confinement must therefore be programmable: the host must be able to observe syscall events and tighten policy live, without re-launching the workload.

2.2. Why Existing Mechanisms Fall Short

Table 1 compares Sandlock against representative isolation mechanisms.

Containers and microVMs are designed for service deployment; their image and namespace machinery can impose deployment overhead disproportionate to a single 50-millisecond Python invocation. gVisor’s user kernel adds compatibility and performance costs for build tools with subtle syscall dependencies. Tools such as firejail and bubblewrap stop short of programmable runtime policies and lack a host-level network ACL. None of these tools natively expose HTTP method/path-level access control, a useful granularity for agent network policy.

2.3. Goals

We extracted five design goals from the workload above:

  • •

    G1 Unprivileged. No root, cgroups, or pre-provisioned namespaces.

  • •

    G2 Low latency. Sub-10 ms startup; near-native runtime overhead.

  • •

    G3 Programmable. The host application can attach a callback that observes syscall events and adjusts policy live, without re-launching the workload.

  • •

    G4 Composable. Multi-stage workloads (planner/executor, mapper/reducer) should pipe data between stages whose confinements differ, so that the capability separation a prompt-injection-resistant decomposition relies on is enforced by the kernel rather than trusted to the LLM’s judgment.

  • •

    G5 Reversible. Filesystem effects can be captured, previewed (--dry-run), and either committed or discarded on exit.

3. Policy Model

Sandlock exposes policy at the level of a command or pipeline stage. A policy describes which host resources the stage may observe, which effects it may produce, and which runtime events the embedding application may inspect before allowing the stage to continue. The model is intentionally smaller than a container configuration: it does not describe images, namespaces, or virtual networks, but the permissions and effects relevant to short-lived agent actions.

3.1. Static Rules and Runtime Decisions

The first distinction is between rules whose verdict is known before the child executes and rules whose verdict depends on syscall-time state. Static rules include readable and writable path prefixes, denied paths, TCP ports, IPC scope, and unconditional syscall denials. Runtime decisions include the resolved destination of a connect, the argument vector of an execve, an HTTP method/host/path, and filesystem effects that should be captured rather than applied directly. Sandlock requires runtime decisions to be made at a point where the relevant syscall can still be allowed, denied, or virtualized; if the decision cannot be made without a race or semantic mismatch, the policy fails closed.

3.2. Policy Domains

Filesystem policy grants read and write capabilities over host paths and can attach an effect action to writes: COMMIT merges captured changes, ABORT discards them, and KEEP leaves them for inspection. Network policy names endpoints by protocol and destination, with optional HTTP rules that further constrain method, host, and path. Resource policy bounds process count, memory, CPU share, and file descriptors. Runtime policy exposes selected syscall events to a host callback (policy_fn), which may allow, deny, audit, or tighten the live policy for the remaining execution.

3.3. Stage Composition

For prompt-injection and data-flow isolation, policies compose across stages rather than only within one process tree. A pipeline is a sequence of stages connected by pipes; each stage has its own filesystem, network, resource, and runtime policy. This lets an agent author express capability separation directly: for example, one stage may read private data but lack network access, while another may call an external API but never receive that data except through an explicit pipe.

4. Sandlock Design

Sandlock maps the policy model onto a split enforcement architecture. It follows one rule throughout: use the earliest enforcement point that can decide a policy soundly. If a decision is known before execve, Sandlock compiles it into Landlock or seccomp-bpf so the kernel enforces it without supervisor involvement. If a decision depends on syscall-time state, such as a resolved network address, an HTTP request, an execve argument vector, or a filesystem effect that should be captured rather than applied, Sandlock routes only that syscall class through seccomp user notification. If neither layer can enforce the policy without a race or a semantic mismatch, Sandlock denies the operation rather than silently weakening the policy.

4.1. The Confinement Pipeline

Confinement is applied in a fixed order after fork(), illustrated in Figure 1. Each step is unprivileged and idempotent. The child blocks before exec() until the parent signals readiness, ensuring the supervisor is attached to the seccomp notification fd before any user code runs.

4.2. Landlock as the Static Spine

Landlock (Salaün, 2017) provides the kernel-enforced “static” layer of policy. Sandlock uses it for constraints whose verdict is independent of syscall-time data: filesystem read/write scope, TCP port scope, and IPC boundaries. When a rule can be expressed this way, common operations are denied or allowed by the kernel without involving the supervisor. Sandlock requires Landlock ABI 6 or later (Linux 6.12+), which supplies filesystem rules (ABI 1), TCP-port rules (ABI 4), and abstract-socket and signal scoping for IPC (ABI 6) (Linux man-pages Project, 2025b); it queries the runtime ABI at startup and refuses to run below this floor rather than silently downgrading to a weaker policy. A check subcommand reports per-feature support (the evaluation kernel, 6.18, provides ABI 7).

4.3. seccomp-bpf as the Filter

After Landlock, Sandlock installs a seccomp-bpf filter that denies syscalls outside Landlock’s scope, routes supervisor-visible syscalls to seccomp user notification (Table 2), and lets the remaining calls proceed in the kernel.

The deny set is conservative by default. Because Landlock already covers filesystem, network, and IPC, the seccomp deny set focuses on syscalls that have no Landlock equivalent.

4.4. The seccomp Notification Supervisor

The dynamic layer of Sandlock is a user-notification supervisor, an async (tokio) task in the parent that dequeues notifications from the seccomp fd and replies with Allow, Deny(errno), or Continue after performing an on-behalf operation. Table 2 summarizes the syscalls handled.

For “acting on behalf” (e.g. to call connect from the child’s address space using a parent-validated address), the supervisor uses pidfd_getfd (Brauner, 2024) to dup the child’s socket into the parent. This is TOCTOU-safe along two axes. The original syscall is never Continued, so the kernel never re-reads the child’s pointer arguments. Where an argument is itself passed by pointer (the destination sockaddr of connect/sendto or the payload of sendmsg), the supervisor reads it once, validates that copy, and issues the operation from its own copy, so a peer thread cannot substitute a different value after validation; each such read is bracketed by SECCOMP_IOCTL_NOTIF_ID_VALID so a read against a task that has exited or been replaced is rejected rather than trusted.

4.5. Network Model

Sandlock uses a single endpoint allowlist that names protocol ×\times destination. Rules may refer to concrete TCP, UDP, or ICMP endpoints, or to TCP ports when host identity is irrelevant. Concrete hostnames are resolved once at sandbox start and pinned for the child process, eliminating DNS-rebinding attacks. Wildcard hostnames are intentionally outside the model: every allowed domain must resolve to a concrete destination that the supervisor can check.

Two enforcement paths implement this model:

Direct path.

If TCP policy depends only on ports and no HTTP rule is active, Landlock enforces TCP-connect at the kernel level with no per-syscall overhead. UDP and ICMP, which Landlock does not cover, fall back to the on-behalf path when allowed.

On-behalf path.

Host-specific rules, HTTP rules, and non-TCP protocols route network syscalls through seccomp notification. The supervisor validates the destination against the resolved allowlist, then either performs the operation on behalf of the child or denies it.

4.6. HTTP/HTTPS Access Control

Sandlock refines endpoint policy with HTTP method, host, and path rules. For intercepted HTTP flows, the supervisor redirects the connection to a local proxy that parses request headers and applies the rule set before forwarding. HTTPS inspection is opt-in because it requires the workload to trust a sandbox CA; otherwise TLS flows are governed only by endpoint policy. HTTP rules also imply the corresponding endpoint permissions, so the policy does not require duplicate host-level configuration.

4.7. Resource Limits Without Cgroups

Without cgroups, traditional resource caps are unavailable. Sandlock reuses the supervisor for accounting:

  • •

    Memory. The supervisor accounts for the virtual address space requested through mmap/mremap/brk/shmget; since resident pages cannot exceed mapped address space, this upper-bounds the footprint of those mappings. On overshoot the process is terminated.

  • •

    Process count. Fork-like syscalls are gated by a counter; thread creation (CLONE_THREAD) is excluded, and excess processes receive EAGAIN.

  • •

    CPU throttling. An optional task issues SIGSTOP/SIGCONT to the child’s process group at a duty cycle proportional to max_cpu.

  • •

    File descriptors use a standard RLIMIT_NOFILE.

The trade-off is that these limits are enforced cooperatively via syscall interception rather than at allocation time inside the kernel, and the memory bound is on address space requested rather than resident pages: growth that bypasses these syscalls (notably main-thread stack growth, which RLIMIT_STACK governs instead) is not counted, and for a pure address-space cap RLIMIT_AS is a simpler kernel-enforced equivalent. We route memory through the supervisor so the same accounting can also virtualize /proc/meminfo. For accidental over-allocation by trusted-but-buggy code these limits suffice; for adversarial workloads aiming to exhaust kernel memory through corner-case syscalls, cgroups remain stronger (Zheng et al., 2026a).

5. Implementation

Sandlock is implemented in ∼\sim7K lines of Rust. Its core library constructs Landlock rules, compiles seccomp filters, runs the async supervisor, and implements the COW, pipeline, and runtime-callback mechanisms. A small command-line frontend and FFI layer expose the same policy model to shell and Python callers.

The remainder of this section describes the implementation details most relevant to safety and portability: the runtime callback boundary, the unprivileged COW backends, and the fork/pipeline runtimes.

5.1. Programmable Policies via policy_fn

The runtime policy hook from Section 3 is implemented as a callback, policy_fn, registered in Rust or Python (Figure 2). The callback receives an Event containing the syscall name, category, PID/PPID, selected network destination fields, and argv for execve. It also receives a Context for live policy updates.

Argv inspection is an observation and policy-tightening signal, not a containment boundary: matching a binary name cannot stop adversarial code, which can rename or reimplement the tool. Containment comes from Landlock, seccomp, and the network ACL; policy_fn adds the ability to detect a phase transition (e.g. setup →\rightarrow untrusted execution) and tighten those kernel-enforced policies live for the rest of the run.

TOCTOU safety.

Per seccomp_unotify(2), after the supervisor returns Continue, the kernel re-reads user-memory pointers. A naive implementation that copies argv into the event and then Continues the syscall would race with a peer thread or process aliasing the same memory. Sandlock avoids this race by keeping path strings out of runtime events: path-based control remains in static Landlock rules, which are TOCTOU-immune, and live path restrictions are represented as policy updates rather than inspected syscall arguments. For argv, which the host may need to inspect, Sandlock freezes every task that could alias the child’s address space before reading arguments or continuing an execve. It enumerates sibling threads and peer processes from /proc and its process index, then stops them with ptrace seize/interrupt requests. Fork-like syscalls (clone, fork, vfork, and clone3) are traced for one ptrace creation event so children are registered before user code runs; for clone3, Sandlock reads flags from the clone-argument structure. If freezing or creation tracking is unavailable (e.g. Yama blocks ptrace), the syscall is denied with EPERM rather than silently relaxed.

The held syscalls, those that block until the callback returns, are execve, connect, sendto, sendmsg, sendmmsg, bind, and openat. Verdicts are: 0/False = allow, True/-1 = deny (EPERM), positive integer = deny with that errno, "audit" = allow + flag.

5.2. Copy-on-Write Workspace Backends

Sandlock provides two COW backends, both unprivileged:

Seccomp COW (default).

The supervisor intercepts filesystem syscalls and redirects writes into an upper directory while resolving reads against upper and lower layers. Directory reads are merged in the supervisor. No mount namespace is needed.

BranchFS.

BranchFS,111https://github.com/multikernel/branchfs/ a dedicated copy-on-write filesystem, captures writes at the filesystem layer rather than through per-syscall interception. Sandlock drives the branch lifecycle with ioctls on a control file (create on entry, commit or abort on exit), and an optional storage quota bounds the branch, surfaced to the child as ENOSPC. Like the seccomp backend it needs no mount namespace in the child, but it keeps the write path inside the filesystem, avoiding a supervisor round-trip per filesystem syscall.

Both backends expose the same effect interface: captured filesystem updates can be committed, discarded, or kept after the sandbox exits.

5.3. COW Fork Runtime

The COW-fork primitive runs initialization once and then forks NN confined workers so they share initialized pages copy-on-write. The runtime captures per-worker output and can pipe merged results to a separately confined reducer. On our setup this sustains ∼\sim1,900 forks/s, enabling map-reduce patterns directly inside an agent process.

5.4. Pipeline Runtime

The stage-composition policy is implemented by connecting sandboxed stages with kernel pipes. Each stage carries its own confinement, allowing the author to separate private-data access, network access, and exposure to untrusted content across different process trees. Figure 3 shows the pattern.

Because each stage is its own Sandlock process tree, a compromise of tr cannot read /opt/data: only the upstream cat could, and its only output channel is the pipe. The same structure enforces the capability boundary that dual-LLM and CaMeL-style decompositions (Willison, 2023; Debenedetti et al., 2025) assume: in the planner/executor split, the stage exposed to untrusted content can be denied network and the stage with network can be denied data access, so a hijacked stage holds no capability it was not granted. Two caveats bound this claim. First, Sandlock bounds stage capabilities, not pipe content: a compromised upstream stage can still shape bytes consumed downstream, so data-provenance defenses (Debenedetti et al., 2025) remain complementary. Second, sound decompositions remain the author’s responsibility: Sandlock kernel-enforces the separation specified, but does not prove that the untrusted-content stage never needs network or that the networked stage never needs private data.

6. Preliminary Evaluation

Setup.

Experiments run on a typical Linux workstation: an AMD Ryzen 5 5500U (6 cores / 12 threads, up to 4.0 GHz) with 8 GB of RAM and NVMe SSD storage, running Pop!_OS 24.04 LTS (an Ubuntu 24.04 derivative) on Linux 6.18 with unprivileged user namespaces enabled. Startup latency is the median of 20 runs (after a warm-up); Redis throughput and tail latency are the median of 5 runs of redis-benchmark (100,000 requests, 50 concurrent clients, 256 B values), and error bars in Figure 4 show ±\pm1 standard deviation. We report bare-metal, Sandlock (default static rules), and Docker for context. We benchmark rootful Docker here, the common deployment and the more favorable baseline for Docker; Table 1 instead compares features against rootless Docker, the unprivileged competitor relevant to Sandlock’s no-root claim.

Effectiveness.

Before measuring overhead, we check that the rules and supervisor actually confine the workload. Table 3 reports deterministic deny/allow cases drawn from the threat model (Section 2); each is run against Sandlock and the observed outcome recorded. Reads and writes outside the granted scope are denied while in-scope access succeeds; a connect to a host outside the endpoint allowlist is refused while an allowlisted endpoint connects; and a stage not granted a private directory cannot read it even though a sibling stage can, the per-stage capability separation the pipeline relies on. Cooperative resource caps also hold: process creation past the limit returns EAGAIN, and an allocation past the memory cap is blocked (the process is terminated rather than over-committing). All ten checks behaved as the policy requires.

Startup overhead.

Figure 4(a) shows /bin/echo wall time. Sandlock adds ∼\sim5 ms over bare metal, dominated by Landlock ruleset construction and seccomp-fd setup; Docker is roughly 44×\times slower due to image and namespace setup.

Throughput.

We benchmark Redis 7 with redis-benchmark -n 100000 (Figure 4(b,c)). Sandlock matches bare-metal throughput on both SET and GET to within run-to-run variance (medians over 5 runs of 75.2k vs. 75.5k rps for SET and 74.2k vs. 73.9k rps for GET), with p99 latency close to bare metal (0.51 vs. 0.49 ms); Docker reaches only ∼\sim76% of bare-metal throughput and has roughly 3×\times the p99 tail latency. Any residual cost from the seccomp filter and the on-behalf bind for the listening socket falls below this workload’s measurement noise.

COW fork.

A 1,000-fork run takes ∼\sim530 ms. This is ∼\sim530 μ\mus/fork, or ∼\sim1,900 forks/s. The cost is dominated by the ptrace creation handshake for policy_fn-safe child registration; with policy_fn disabled, fork-rate increases by another ∼\sim30%.

Network on-behalf.

The Landlock direct path (port-only policy) leaves traffic in the kernel and adds no per-syscall overhead. The on-behalf path is different: each send is mediated by the supervisor, adding a fixed per-message cost. For a 256 B request/response echo with both endpoints sandboxed, the median round-trip rises from 20 μ\mus on bare metal to 55 μ\mus (∼\sim35 μ\mus added; p99 33 μ\mus vs. 114 μ\mus), whereas Docker, whose established connections stay in the kernel, matches bare metal (19 μ\mus median). This fixed cost is immaterial for the network workloads agents actually run, such as LLM API calls measured in 100 ms–10 s.

Compatibility.

Common build tools run under Sandlock once their scratch paths are granted: we ran python3, make, node, and pytest unmodified; Python native-extension imports also worked. The main tuning is to grant each tool its temporary directories (e.g. /tmp for compiler intermediates, /dev/null). A systematic compatibility study across the long tail of agent tooling is future work.

7. Related Work

Unprivileged Linux primitives.

Sandlock is built on top of Landlock (Salaün, 2017), seccomp-bpf (Drewry and Edge, 2024), and seccomp user notification (Kerrisk, 2024), with pidfd_getfd (Brauner, 2024) for safe fd transfer. Capsicum (Watson et al., 2010) is the closest spiritual ancestor: a capability-based unprivileged sandbox, originally for FreeBSD, later partially ported. Container runtimes embody the same split at a coarser grain: runc and Docker can forward selected syscalls to a userspace agent through seccomp notification (Kerrisk, 2024; Open Container Initiative, 2023). Sandlock’s contribution is therefore not the use of notification (which is its kernel-documented purpose) but how the primitives are organized: a single policy surface spanning filesystem, network, IPC, and resources; a TOCTOU-safe runtime hook that exposes execve argv to host code; and unprivileged copy-on-write plus pipeline composition built on the same supervisor. Among unprivileged process sandboxes, nsjail (Święcki, 2024) and minijail (The ChromiumOS Authors, 2024) compose namespaces with seccomp-bpf, and Rust libraries such as birdcage (Phylum, 2023) provide embeddable static sandboxing APIs; these enforce static policy but expose neither a runtime callback nor host/HTTP-level network rules. BPF-LSM sandboxes such as bpfbox (Findlay et al., 2020) provide programmable policy through eBPF, but loading LSM BPF generally requires CAP_BPF or root (Linux man-pages Project, 2025a), which Sandlock avoids. The classic hazards of syscall interposition (argument races and indirect paths) were catalogued by Garfinkel (Garfinkel, 2003); Sandlock’s TOCTOU handling (Section 5.1) is designed against exactly these.

Container and microVM sandboxes.

Docker (Merkel, 2014) and rootless containers compose namespaces (Kerrisk, 2013), cgroups (Heo, 2015), and seccomp profiles into reusable images; gVisor (Google, 2024) adds a user-mode kernel; Firecracker (Agache et al., 2020) and Catalyzer (Du et al., 2020) reduce microVM startup. These target multi-tenant service hosting, where image immutability and strong kernel isolation outweigh the startup cost. Sandlock targets the orthogonal workstation use case where startup cost matters and root is unavailable.

Process-internal isolation.

lwCs (Litton et al., 2016) introduce in-process protection domains for fast context switching; Dune (Belay et al., 2012) uses VT-x for user-level kernels. These provide stronger isolation within a process; Sandlock is concerned with confining a different process tree.

Agent systems.

MVVM (Yang et al., 2024b) deploys agents in WebAssembly enclaves; Kgent (Zheng et al., 2024) explores LLM-driven kernel extensions; AgentCgroup (Zheng et al., 2026a) uses cgroups to characterize and control the OS resources AI agents consume; Agentry (Wang, 2025) mediates agent-to-external communication through a gateway. Sandlock is complementary: it confines the local execution side of an agent stack while gateways like Agentry mediate the communication side.

Prompt injection.

The dual-LLM pattern (Willison, 2023) and CaMeL (Debenedetti et al., 2025) defend against prompt injection by structurally separating the component exposed to untrusted content from the component holding sensitive capabilities. These works define the decomposition; Sandlock provides an unprivileged, kernel-enforced substrate on which the separated stages run, via its pipeline operator and heterogeneous per-stage confinements.

COW and speculation.

OverlayFS (Brown, 2016) and UnionFS (Wright et al., 2006) provide kernel-level COW; Sandlock’s seccomp COW is a userspace alternative that requires no mount namespace. MBOX (Kim and Zeldovich, 2013) layered a sandbox filesystem over a process via ptrace interposition with a commit step, much as Sandlock’s seccomp COW captures writes for commit or abort; Sandlock uses seccomp notification rather than ptrace and shares one supervisor across its filesystem, network, and runtime-policy paths. Speculator (Nightingale et al., 2005) and TxOS (Porter et al., 2009) explored OS-level transactions and output gating; Sandlock’s --dry-run mode is a much narrower point in this design space, restricted to filesystem effects. Recent agent-oriented work revisits these primitives: Fork, Explore, Commit (Wang and Zheng, 2026) proposes speculative agent exploration with commit/rollback, matching Sandlock’s COW-fork and workspace effects; ACRFence (Zheng et al., 2026b) shows that checkpoint-based rollback can itself be attacked.

8. Discussion and Future Work

Threat-model boundaries.

Sandlock is unprivileged by design and therefore inherits the limits of unprivileged Linux. Kernel-level vulnerabilities, side channels, and resource-exhaustion attacks against shared kernel structures are out of scope. For multi-tenant adversarial workloads, a microVM remains the right answer.

Integration.

Sandlock supports two retrofit modes. In wrap-the-child, the host forks, confines, and execs each command an agent already spawns, a drop-in for the shell and tool calls an agent issues, requiring no change to the agent loop. In self-confinement, the static Landlock layer is applied to the calling process in place (a confine() call), so an agent runtime can drop its own filesystem and network privileges after startup; the dynamic supervisor, which needs a separate parent, remains a wrap-the-child facility. MCP tool servers are a natural fit for the first mode: a long-running server confines each tool invocation as a child under a per-tool policy, isolating untrusted tool execution without re-architecting the agent.

Beyond the local host.

HTTP-level ACLs are a useful step, but agentic workloads increasingly span tool servers (MCP) and remote APIs. We are exploring an integration in which Sandlock cooperates with an Agentry-style (Wang, 2025) gateway: the local sandbox enforces method/host/path; the gateway enforces semantic and per-tenant policies on the same flow.

Effect rollback.

Sandlock captures filesystem effects but not network effects. A natural extension is to buffer outbound HTTP requests through the proxy and release them only after the host commits the sandboxed stage, mirroring the speculator-style output gating literature (Nightingale et al., 2005) for the agent setting.

Composition with branching filesystems.

Sandlock’s BranchFS backend already gives a single sandbox a copy-on-write branch, but its pipeline and COW-fork primitives do not yet exploit branching. A natural next step is to run each pipeline stage and each COW-fork worker on its own branch and merge the results on commit, giving multi-branch agents end-to-end isolation across both filesystem and process state.

9. Conclusion

We presented Sandlock, an unprivileged process sandbox tailored to the AI-agent workstation. Its central design is a split enforcement model: static policy is compiled into Landlock and seccomp-bpf, while a narrow seccomp-notification supervisor handles runtime-dependent decisions and virtualized effects. This structure gives Sandlock a programmable runtime policy surface (policy_fn), HTTP-level ACLs, copy-on-write workspaces, COW-fork map/reduce, and pipeline composition while preserving ∼\sim5 ms startup overhead and near-native Redis performance. Sandlock is open source at https://github.com/multikernel/sandlock.
